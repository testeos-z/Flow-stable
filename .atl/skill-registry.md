## MCP Tools — flow-validation (flowise-mcp-server)

Las siguientes herramientas están disponibles en el MCP `flowise-mcp-server`:

### Validación (todos los agentes pueden usar)

| Tool                   | Propósito                                                     |
| ---------------------- | ------------------------------------------------------------- |
| `validate_flow_data`   | Valida estructura JSON (nodes, edges, viewport)               |
| `validate_flow_graph`  | Valida conectividad (orphans, ciclos, desconectados)          |
| `fix_flow_data`        | Repara issues comunes antes de guardar                        |
| `full_flow_validation` | Validación completa (schema + graph + fix)                    |
| `validate_chatflow`    | Valida flowData sin guardar (revisa credenciales, estructura) |

### Testing (solo flow-ing)

| Tool                | Propósito                                    |
| ------------------- | -------------------------------------------- |
| `test_chatflow`     | Smoke test + integration test en un chatflow |
| `diagnose_chatflow` | Diagnóstico de campos faltantes en DB        |
| `repair_chatflow`   | Repara chatflow inyectando defaults en DB    |

### Credentials (todos los agentes pueden usar)

| Tool                 | Propósito                                    |
| -------------------- | -------------------------------------------- |
| `list_credentials`   | Lista credenciales disponibles (type → UUID) |
| `resolve_credential` | Resuelve tipo de credencial a UUID           |

**Reglas**:

-   SIEMPRE usar `validate_chatflow` antes de crear/actualizar
-   SOLO `flow-ing` puede ejecutar `test_chatflow` y operaciones de escritura
-   `flow-architect` usa `list_credentials` y `resolve_credential` para obtener UUIDs

## Compact Rules

### alejandria-architecture

-   Alejandria = data/knowledge layer only — NO agents, NO LLM orchestration
-   Expose: `search()`, `getDocument()`, `getEdgeData()`, `getStatic()`
-   Patterns: Factory (MCP clients), Repository (vector/static), Strategy (per-MCP), Decorator (cache)
-   Do NOT couple to Supabase directly — abstract behind interfaces
-   Consumers: Flowise flows via MCP tools

### flowise-node-reference

-   **ACCESS RESTRICTED**: Solo flow-ing y flow-architect pueden cargar este skill
-   Always verify node names against the 302-node catalogue before using
-   Check credential requirements per node (100 credential types)
-   For agent flows: prefer Sequential Agents over basic LLM chains for multi-step logic
-   Before creating/editing flows via API, load `references/04-flowdata-schema.md` to understand the JSON structure

### flow-architect (a2a-lab) — READ-ONLY ORQUESTADOR

-   **NO puede escribir en Flowise** — solo leer, diseñar, orquestar
-   Cuando necesita crear un flow: diseña → delega a specialists → ensambla → delega a flow-ing
-   Three flows in sequence: FACTUM → ÁGORA → POLITEIA
-   Each flow is independent, consumes output of the previous
-   Supabase is the shared state store between flows

### flow-ing — ÚNICO EXECUTOR

-   **Único agente con permiso de escritura** en Flowise
-   Ejecuta testing pipeline (5 stages) antes de cualquier escritura
-   Si tests fallan → reporta errores, NO guarda
-   Comandos disponibles: `/flow-diagnose <chatflowId>`

### Node Specialists (determinísticos, NO LLM)

-   Cada specialist tiene Zod schema basado en golden templates
-   Validan: estructura completa, template syntax, credentials UUID, capabilities
-   Se ejecutan en paralelo cuando no hay dependencias

### javascript-testing-patterns

-   Project uses Jest (server/components) + Vitest (flowise-mcp-server)
-   Use `ts-jest` for server-side TypeScript tests
-   Strict TDD Mode: ACTIVE — write tests before implementation
